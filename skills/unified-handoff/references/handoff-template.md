# Handoff Template Guidance

Use the scaffold generated by `create`; do not maintain a separate hand-written template copy.

## Required content

- Objective: exact outcome and why it matters
- Current State: working, broken, incomplete, and precise stopping point
- Immediate Next Steps: ordered, executable actions with file/command/verification targets

## High-value sections

### Decisions

Record alternatives, evidence, and rationale—not only the chosen result.

### Attempts and Failures

Use columns:

```text
Attempt | Evidence/Result | Why It Failed | Retry Condition | Do Not Repeat
```

A failed approach marked `Do Not Repeat: Yes` may only be retried after its retry condition is satisfied.

### Evidence and Verification

Separate commands actually run from commands merely inferred by the scaffold. Record pass/fail status, relevant output summaries, measurements, error summaries, and unverified items.

### User Requirements and Corrections

Requirements capture scope and constraints. Corrections receive their own section because they supersede earlier agent assumptions.

### Knowledge Status

Every material uncertain claim should be classified as Verified Fact, Agent Inference, or Unverified Assumption.

## Content rules

- Prefer repository-relative paths and line/symbol references.
- Link plans, ADRs, issues, commits, and files instead of pasting full diffs.
- Never invent evidence or test results.
- Never include secret values or environment-variable values.
- Use `N/A - <reason>` only when a non-blocking section truly does not apply.
